## Decisions & Rationale

1. **Ki=0 (proportional+derivative only)** — the vault never cools below setpoint because NAS heat generation exceeds fan cooling capacity. Any Ki>0 accumulates indefinitely. Validated across three failed iterations with overnight data.
2. **Negative PID gains for cooling** — `simple-pid` uses `error = setpoint - measured`. Cooling applications need negative Kp/Ki/Kd so that hot produces positive output.
3. **Setpoint=122°F from InfluxDB** — 6-month idle cluster centers at 122-124°F. Setpoint at cluster floor means fan off at equilibrium, ramps for load spikes.
4. **MQTT v5 over v3.1.1** — will delay interval, session expiry, and message expiry are v5-only features. EMQX v5 and paho-mqtt 2.1 both support it.
5. **LWT with 30s will delay** — suppresses false offline alerts during systemd restarts (10s) and WiFi blips.
6. **venv on Pi, conda on workstation** — Pi Zero W is armv6l which conda doesn't support.
7. **Single config file** — `pi/config.yaml` is the sole source of truth for controller, debug client, tests, and HA entity references.

## Reusable Insights

- **HA automations only fire on `last_changed`, not `last_updated`.** Sensors reporting the same quantized value don't trigger state-based automations. Fix: add `time_pattern` as a parallel trigger.
- **For cooling PIDs with `simple-pid`: negate all gains.** The library's error convention (`setpoint - measured`) means positive gains = heating.
- **InfluxDB HA schema: `_measurement` is the unit string (e.g., "°F"), `_field` is "value", `entity_id` is without the domain prefix.**
- **To create HA helpers programmatically, edit `.storage/input_number` (JSON) and restart. The REST API does not expose helper creation.**
- **MQTT v5 Will Delay Interval prevents brief reconnects from triggering false offline alerts — critical when systemd auto-restarts services.**
- **Temperature sensor quantization (1°C) propagates through the PID as ~22.5% speed steps with Kp=-12.5. The fan is continuous but the dashboard looks stepped.**
